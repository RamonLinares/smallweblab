# Small Web Lab

Static product-lab site for `smallweblab.com`, generated by Quiremark.

This repository owns the site content, configuration, uploaded imagery,
prototype source snapshots, deploy configuration, and generated `out/` output.
Quiremark owns the compiler, templates, admin UI/server code, and publish
logic through the package dependency.

## Project Shape

- `content/settings.json`: site identity, SEO, analytics, categories, widgets, and theme choice
- `content/posts/`: published Markdown entries for products and lab prototypes
- `content/images/`: site-owned images copied into `out/content/images/` on publish
- `lab/`: synced prototype microsite source snapshots
- `scripts/sync_prototypes.py`: mirrors configured prototype repos into `lab/<slug>/`
- `scripts/copy_site_static.js`: copies site-owned static prototype routes into `out/`
- `.github/workflows/sync-prototypes.yml`: keeps prototype snapshots current
- `.github/workflows/deploy-pages.yml`: builds `out/` and deploys it to GitHub Pages
- `favicon.ico`: legacy site favicon asset
- `out/`: generated static website output, ignored by git

There is no vendored Quiremark engine source in this repo.

## Local Setup

```bash
npm install
npm run build
python3 -m http.server 4173 --directory out
```

Then open `http://localhost:4173/`.

## Publishing

The publish package script calls the Quiremark CLI from the dependency:

```bash
npm run publish
```

which runs:

```bash
quiremark publish --site-root .
```

For Cloudflare Pages, use:

- Install command: `npm ci`
- Build command: `npm run build`
- Build output directory: `out`

For GitHub Pages, `.github/workflows/deploy-pages.yml` builds the site on pushes
to `main` and deploys the generated `out/` artifact with GitHub Actions. In the
repository settings, Pages should use **GitHub Actions** as its source.

The compiler writes HTML, category pages, post pages, search assets, RSS,
`robots.txt`, `sitemap.xml`, `llms.txt`, and `llms-full.txt`.

`npm run build` runs the Quiremark publish command and then copies the
site-owned `lab/` prototype routes and `favicon.ico` into `out/`.

`QUIREMARK_DATA_DIR` is the Quiremark data-directory environment variable. This
external-site workflow normally uses `--site-root .`; set `QUIREMARK_DATA_DIR`
only when running Quiremark server/admin workflows that need a persistent data
directory.

## Content Workflow

Add or edit posts in `content/posts/`.

Each post uses front matter:

```yaml
---
title: "Example"
slug: "example"
description: "Short summary for cards and SEO."
date: "2026-05-21"
category: "products"
tags: ["Static Site", "Product"]
coverImage: "/content/images/example.jpg"
draft: false
---
```

Site-level settings live in `content/settings.json`. The current public

## Prototype Sync

`lab/catalog.json` remains the source of truth for synced prototype source
snapshots.

`scripts/sync_prototypes.py` mirrors each configured source repository into
`lab/<slug>/`.

Current prototype routes referenced by site content:

- `ping-pong-3d` -> `/lab/ping-pong-3d/`
- `asteroids-3d` -> `/lab/asteroids-3d/`
- `rally-rush` -> `/lab/rally-rush/`
- `meeting-notes` -> `/lab/meeting-notes/`

Do not hand-edit synced prototype folders unless you intend to replace the next
sync result.

## Branch Policy

This repo is the live deploy source for `smallweblab.com`.

- If the user says `push it`, `ship it`, `make it live`, or similar, default to updating `main`.
- Do not leave lab additions only on a side branch or draft PR unless the user explicitly asks for PR workflow.
- Use a PR branch only when the user asks to review changes before merge.
- If a side branch already contains the desired change, merge it into `main` before calling the task done.
